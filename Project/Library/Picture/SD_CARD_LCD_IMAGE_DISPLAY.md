# SD Card LCD Image Display

## 1. 概述

在嵌入式系统中，通过 `SD` 卡在 `LCD` 屏幕上显示图片，通常不是简单地“读取文件然后显示”，而是一个完整的数据处理链路：

```text
SD 卡
  -> 文件系统读取
  -> 图片格式识别
  -> 图片数据解析或解码
  -> 像素格式转换
  -> LCD 刷屏显示
```

在这个流程中，真正送到 `LCD` 的并不是 `BMP`、`JPEG`、`PNG` 这些文件本身，而是已经整理成屏幕可以识别的像素流，例如 `RGB565`。

因此，使用 `SD` 卡显示图片时，需要同时解决以下几个问题：

- 如何从 `SD` 卡读取图片文件
- 如何识别图片类型
- 如何把图片数据转换为 LCD 需要的像素格式
- 如何在 RAM 很小的情况下完成显示

## 2. 系统组成

一个典型的 `STM32 + SD 卡 + LCD` 显示系统，通常由以下几个部分组成。

### 2.1 SD 卡接口层

`SD` 卡通常通过以下方式连接：

- `SDIO` / `SDMMC`
- `SPI`

其中：

- `SDIO/SDMMC` 速度较高，更适合图片和文件频繁读取
- `SPI` 接线简单，但吞吐量通常更低

如果图片较大，或者需要切换多张图片，优先考虑 `SDIO`。

### 2.2 文件系统层

大多数嵌入式工程会使用 `FATFS` 读取 `SD` 卡中的文件。

常见操作包括：

- 挂载文件系统
- 打开图片文件
- 读取文件头
- 按块或按行读取数据

例如：

- `f_mount()`
- `f_open()`
- `f_read()`
- `f_lseek()`
- `f_close()`

### 2.3 图片解析层

这一层负责判断当前文件属于哪种格式，并执行相应处理：

- `BMP`：读取文件头、宽高、位深，按行读取像素
- `JPEG`：调用 JPEG 解码器
- `PNG`：调用 PNG 解码器
- `RGB565/RAW`：按预定义宽高和像素格式直接读取

### 2.4 LCD 驱动层

这一层负责将像素输出到屏幕，常见操作包括：

- 设置显示窗口
- 写入单个像素
- 连续写入一行像素
- 整块区域刷新

对于彩色 LCD，通常目标格式为：

- `RGB565`

因此，很多显示流程的核心工作就是“把文件中的图像数据变成 `RGB565`”。

## 3. 通过 SD 卡显示图片的整体流程

一般可按以下步骤实现：

1. 初始化 `SD` 卡接口
2. 挂载 `FATFS`
3. 打开指定图片文件
4. 根据文件头或扩展名判断图片类型
5. 解析图片宽度、高度、像素深度
6. 选择合适的解码方式
7. 逐行或逐块读出像素
8. 转换为 `LCD` 所需格式
9. 写入 LCD 显示区域
10. 文件读取结束后关闭文件

如果 MCU 内存有限，不建议一次性将整张图片全部读入内存，而应采用：

- 逐行读取
- 分块读取
- 边读边解码边显示

## 4. 各种常见图片类型的实现路径

不同图片格式在嵌入式中的处理方式差异很大。核心区别在于：文件结构是否简单，运行时解码开销是否可接受。

### 4.1 BMP 图片

`BMP` 是在嵌入式中最容易从 `SD` 卡读取并显示的格式之一。

原因：

- 文件头结构清晰
- 宽高、位深容易获取
- 支持逐行读取
- 不依赖复杂压缩解码

典型流程：

1. 打开 `BMP` 文件
2. 读取文件头和信息头
3. 获取图片宽度、高度、位深、像素数据偏移
4. 根据位深选择处理方式，例如：
   - `16bit BMP`
   - `24bit BMP`
   - `32bit BMP`
5. 定位到像素起始位置
6. 每次读取一行像素
7. 将像素转换为 `RGB565`
8. 写入 LCD

需要注意：

- 很多 `BMP` 采用“自底向上”存储
- 每行可能带字节对齐填充
- `24bit BMP` 通常是 `BGR` 排列，不是 `RGB`

优点：

- 实现简单
- 适合小内存 MCU
- 非常适合从 `SD` 卡逐行读取显示

缺点：

- 文件体积通常较大
- 不适合大量照片资源长期存储

### 4.2 JPEG 图片

`JPEG` 适合存储照片类图片，优点是体积小，但在 MCU 上直接显示时，问题主要在解码复杂度。

典型流程：

1. 从 `SD` 卡读取 `JPEG` 文件
2. 调用 JPEG 解码库解析压缩数据
3. 输出像素块或行缓冲
4. 转换为 `RGB565`
5. 刷新到 LCD

优点：

- 文件小，节省 `SD` 卡空间
- 适合相册、照片浏览

缺点：

- 解码需要较高 CPU 性能
- 需要较大工作缓冲区
- 软件实现复杂度高

适用场景：

- `STM32F4/F7/H7` 等性能较强平台
- 带足够 RAM 的系统
- 已集成 JPEG 解码库或硬件加速单元

对于普通小型 MCU，如果只是显示固定图片，不建议运行时直接解码 `JPEG`，更实用的做法通常是：

- 在 PC 端先转成 `RGB565`
- 再存入 `SD` 卡供 MCU 直接读取显示

### 4.3 PNG 图片

`PNG` 常用于图标、界面素材、透明背景资源。但在 MCU 上直接从 `SD` 卡读取并显示时，难点在于：

- 存在无损压缩
- 可能有 Alpha 通道
- 解码步骤多于 `BMP`

典型流程：

1. 从 `SD` 卡读取 `PNG` 文件
2. 调用 PNG 解码库展开像素数据
3. 必要时处理透明通道
4. 转换成 `RGB565`
5. 刷新 LCD

优点：

- 适合 UI 图标与带透明背景素材
- 文件通常比 BMP 更小

缺点：

- 解码复杂
- RAM 压力较大
- 透明混合会增加运算量

适用场景：

- 带 GUI 框架的较强平台
- 对 UI 效果要求较高的系统

对大部分 `STM32 + 普通 LCD` 工程，更推荐：

- 设计阶段用 `PNG`
- 发布前离线转成 `RGB565` 或轻量压缩格式

### 4.4 RGB565 / RAW 图片

这类格式不是通用桌面图片格式，而是面向嵌入式显示优化的原始像素数据。

常见做法：

- 在 PC 端把图片转换为 `RGB565`
- 存成二进制文件放入 `SD` 卡
- MCU 运行时直接读出并写入 LCD

典型流程：

1. 打开 `RGB565` 原始文件
2. 已知图片宽高
3. 每次从 `SD` 卡读取一行或若干像素
4. 直接送到 LCD

优点：

- 显示最快
- MCU 逻辑最简单
- 很适合逐行显示

缺点：

- 文件不自描述，通常需要额外记录宽高和格式
- 文件体积比压缩格式大

这是嵌入式中最务实、最常见的 `SD` 卡图片显示方案之一。

## 5. 各类图片文件格式中的数据如何存储

这一节重点解释：图片文件内部到底保存了哪些字段，这些字段在嵌入式 LCD 显示时为什么重要。

### 5.1 BMP 文件里的数据如何存储

`BMP` 最适合入门，因为它的结构最直观。一个典型 `BMP` 文件通常由三部分组成：

```text
BMP 文件
  -> 文件头 File Header
  -> 信息头 Info Header
  -> 像素数据 Pixel Data
```

最常见的结构如下：

```text
偏移  大小   含义
0x00   2      文件标志，通常是 'B' 'M'
0x02   4      文件总大小
0x06   4      保留字段
0x0A   4      像素数据起始偏移
0x0E   4      信息头大小
0x12   4      图片宽度
0x16   4      图片高度
0x1A   2      颜色平面数，通常为 1
0x1C   2      位深，例如 16、24、32
0x1E   4      压缩方式
0x22   4      像素数据大小
```

其中嵌入式最关心的字段是：

- 文件标志：判断是不是 `BMP`
- 像素数据偏移：决定从哪里开始读像素
- 宽度和高度：决定 LCD 显示尺寸
- 位深：决定每个像素占多少字节
- 压缩方式：决定能不能直接按像素读

例如：

- `24bit BMP`：每个像素 3 字节，顺序通常是 `B G R`
- `16bit BMP`：每个像素 2 字节，可能是 `RGB565` 或近似格式
- `32bit BMP`：每个像素 4 字节，通常还带透明信息

#### BMP 文件头十六进制示意

下面给一个简化的 `24bit BMP` 文件头示意。这里不是完整真实文件，只是为了帮助理解“偏移”和“字段”如何对应。

```text
偏移    十六进制数据
0x00    42 4D
0x02    36 84 03 00
0x06    00 00 00 00
0x0A    36 00 00 00
0x0E    28 00 00 00
0x12    40 01 00 00
0x16    F0 00 00 00
0x1A    01 00
0x1C    18 00
0x1E    00 00 00 00
0x22    00 84 03 00
```

可以把它理解为：

- `42 4D`：表示这是一个 BMP 文件
- `36 84 03 00`：表示文件总大小
- `36 00 00 00`：表示像素数据从偏移 `0x36` 开始
- `28 00 00 00`：表示信息头大小为 40 字节
- `40 01 00 00`：表示宽度是 320
- `F0 00 00 00`：表示高度是 240
- `01 00`：颜色平面数，固定为 1
- `18 00`：表示位深是 24 位
- `00 00 00 00`：表示不压缩
- `00 84 03 00`：表示像素数据区域大小

#### BMP 按偏移读取字段示例

下面是 MCU 侧最常用的几个偏移位置：

| 偏移 | 长度 | 字段名 | 含义 | MCU 为什么要读它 |
| --- | --- | --- | --- | --- |
| `0x00` | 2 | Signature | 文件标志，通常是 `BM` | 先确认是不是 BMP |
| `0x02` | 4 | File Size | 文件总大小 | 可用于一致性检查 |
| `0x0A` | 4 | Pixel Data Offset | 像素起始偏移 | 确定从哪里开始读像素 |
| `0x0E` | 4 | DIB Header Size | 信息头大小 | 判断头部结构类型 |
| `0x12` | 4 | Width | 图片宽度 | 决定一行像素数 |
| `0x16` | 4 | Height | 图片高度 | 决定总行数以及显示方向 |
| `0x1C` | 2 | Bit Count | 位深 | 判断每个像素占几个字节 |
| `0x1E` | 4 | Compression | 压缩方式 | 判断是否能直接读像素 |

#### 小端存储是什么意思

BMP 头里的多字节整数通常采用“小端存储”，即：

- 低字节放前面
- 高字节放后面

例如：

```text
36 00 00 00
```

这不是十六进制的 `0x36000000`，而是小端表示的：

```text
0x00000036
```

也就是十进制 `54`。

再例如：

```text
40 01 00 00
```

按小端解释后是：

```text
0x00000140 = 320
```

所以 MCU 代码里经常会写类似这种逻辑：

```text
value = buf[0]
      | (buf[1] << 8)
      | (buf[2] << 16)
      | (buf[3] << 24)
```

原因就是要把小端字节重新拼成整数。

#### MCU 读取 BMP 的顺序示例

在 STM32 里读取 BMP 时，常见顺序通常是：

1. 先读前 14 字节文件头
2. 判断前两个字节是不是 `BM`
3. 读信息头
4. 取出宽度、高度、位深、压缩方式
5. 根据像素偏移跳到像素区
6. 按行读取像素数据
7. 把 `BGR` 转成 `RGB565`
8. 每读一行就写到 LCD

这也是为什么 `BMP` 很适合在嵌入式里做第一版图片显示。

#### BMP 像素数据如何存储

BMP 的像素通常按“行”存储，但要注意两个关键点：

1. 很多 BMP 是“自底向上”存储
意思是文件里先存的是图像最底下一行，再往上存。  
所以如果图片高度是正数，MCU 显示时往往要反着算行号。

2. 每行通常按 4 字节对齐
例如：

- 宽度 1 像素、24 位图：理论上 3 字节
- 但 BMP 一行会补齐到 4 字节
- 宽度 2 像素：理论上 6 字节，也会补到 8 字节

这就是为什么读取 BMP 时不能简单地认为：

```text
一行字节数 = 宽度 * 每像素字节数
```

而应该按对齐后计算。

#### 为什么 BMP 对嵌入式友好

因为 MCU 可以这样处理：

1. 读文件头
2. 得到宽高和位深
3. 定位像素起点
4. 每次只读一行
5. 把 `BGR` 转成 `RGB565`
6. 立刻送到 LCD

不需要先把整张图解压到 RAM。

### 5.2 JPEG 文件里的数据如何存储

`JPEG` 和 `BMP` 最大的区别是：`JPEG` 文件里保存的不是“直接可显示的像素数组”，而是压缩后的编码数据。

它的结构不是固定长度头，而是由很多“段”组成，每一段前面都有标记符 `Marker`。

常见结构可以简化理解为：

```text
SOI -> APP0 -> DQT -> SOF -> DHT -> SOS -> 压缩图像数据 -> EOI
```

常见段含义：

- `SOI`：图像开始
- `APP0`：应用信息，例如 JFIF
- `DQT`：量化表
- `SOF`：图像宽度、高度、颜色分量等基本信息
- `DHT`：哈夫曼表
- `SOS`：扫描开始，后面跟压缩后的图像数据
- `EOI`：图像结束

嵌入式最关心的基础信息通常在 `SOF` 段里，例如：

- 宽度
- 高度
- 精度
- 分量数

#### JPEG 数据为什么不能直接刷 LCD

因为 `JPEG` 文件中的主体数据不是像素值，而是经过以下处理后的结果：

- 颜色空间变换
- 分块
- DCT 变换
- 量化
- 哈夫曼编码

所以 MCU 拿到 `JPEG` 文件后，不能像 `BMP` 那样“直接读像素”，而必须：

1. 找到各个 marker 段
2. 读出量化表和哈夫曼表
3. 对压缩比特流做解码
4. 逆量化、逆变换
5. 重新还原成像素
6. 再转成 `RGB565`

这也是 `JPEG` 运行时解码复杂、耗 CPU 和 RAM 的根本原因。

### 5.3 PNG 文件里的数据如何存储

`PNG` 的结构比 `BMP` 复杂，但比 `JPEG` 更规则。它由一个固定签名加多个 `Chunk` 组成。

典型结构：

```text
PNG Signature
  -> IHDR
  -> PLTE（可选）
  -> IDAT（一个或多个）
  -> IEND
```

每个块通常由这几部分组成：

```text
长度 Length
类型 Type
数据 Data
校验 CRC
```

最关键的块如下：

- `IHDR`：保存宽度、高度、位深、颜色类型等
- `PLTE`：调色板，可选
- `IDAT`：真正的图像压缩数据，可能有多个
- `IEND`：文件结束

#### PNG 中哪些字段最重要

在 `IHDR` 中，嵌入式最关心：

- 宽度 `Width`
- 高度 `Height`
- 位深 `Bit Depth`
- 颜色类型 `Color Type`
- 压缩方法
- 滤波方法
- 交错方式

这些字段决定了解码器后续怎样解释 `IDAT` 中的数据。

#### PNG 数据为什么也不能直接刷 LCD

因为 `IDAT` 里存的是经过压缩和滤波后的图像数据，不是原始像素。

MCU 显示 PNG 通常要做：

1. 拼接所有 `IDAT` 数据
2. 做解压缩
3. 对每一行执行反滤波
4. 得到真正像素值
5. 如果有透明通道，还要处理 Alpha
6. 再转换成 `RGB565`

所以 `PNG` 虽然适合做 UI 图标资源，但运行时直接解码通常比 `BMP` 麻烦得多。

### 5.4 RGB565 / RAW 数据如何存储

`RGB565/RAW` 在嵌入式里最直接，因为它往往没有复杂文件格式，存的就是像素本身。

最简单的情况可以理解为：

```text
像素0  像素1  像素2  ...
每个像素 2 字节
顺序通常按从左到右、从上到下排列
```

如果一张图宽 `W`、高 `H`，那么总数据大小通常就是：

```text
W * H * 2 字节
```

例如一张 `100 x 50` 的 `RGB565` 图片：

```text
100 * 50 * 2 = 10000 字节
```

#### RGB565 文件为什么最快

因为 MCU 基本不需要解码：

1. 已知宽高
2. 每次读出若干字节
3. 直接送入 LCD

这几乎就是“文件数据 = LCD 像素数据”。

#### RGB565 的问题是什么

它通常不自描述，也就是说文件本身未必告诉你：

- 宽度是多少
- 高度是多少
- 是不是 `RGB565`
- 是从上到下还是从下到上

因此实际工程里常见做法有两种：

1. 文件名约定
例如：

```text
logo_240x135.rgb565
```

2. 额外配套元数据
例如用：

- 配套 `.txt/.json` 文件
- 自定义小文件头
- 固定资源表

这也是 `RGB565` 虽然显示最快，但管理上不如 `BMP/PNG/JPEG` 通用的原因。

### 5.5 几种格式的存储方式对比

| 格式 | 宽高信息是否内置 | 像素是否直接可用 | 是否压缩 | MCU 直接显示难度 |
| --- | --- | --- | --- | --- |
| `BMP` | 是 | 基本可直接按行读取 | 通常无压缩或简单压缩 | 低 |
| `JPEG` | 是 | 否 | 有损压缩 | 高 |
| `PNG` | 是 | 否 | 无损压缩 | 高 |
| `RGB565/RAW` | 通常否 | 是 | 通常无压缩 | 很低 |

### 5.6 为什么理解文件内部结构很重要

在嵌入式 LCD 项目里，理解这些文件格式内部结构的价值主要有三点：

1. 知道 MCU 该读哪里
例如：

- `BMP` 要先找到像素偏移
- `JPEG` 要先找到 `SOF/SOS`
- `PNG` 要先解析 `IHDR/IDAT`

2. 知道 LCD 拿到的是“原始像素”还是“压缩数据”
如果拿到的不是原始像素，就不能直接刷屏。

3. 知道资源该在哪一端处理
一般策略是：

- 复杂格式尽量在 PC 端处理
- MCU 端尽量只做轻量解析或直接显示

## 6. 小内存 MCU 的三种显示策略

在 MCU RAM 很有限的情况下，显示图片时通常不能把整图完整载入内存。常见策略如下。

### 6.1 整图载入

方式：

- 整个文件读入 RAM
- 解析后一次性显示

优点：

- 控制简单
- 适合小图标和缩略图

缺点：

- 大图容易耗尽 RAM

适用场景：

- 小尺寸图片
- 系统 RAM 较大

### 6.2 逐行读取

方式：

- 每次只读取一行像素数据
- 转换后立即写入 LCD

优点：

- RAM 占用小
- 非常适合 `BMP` 和 `RGB565`

缺点：

- 文件读取次数较多
- 程序控制略复杂

适用场景：

- 中低端 MCU
- 中大尺寸图片显示

### 6.3 边读边解码边显示

方式：

- 从 `SD` 卡读一部分压缩数据
- 解码出像素后立即刷新 LCD

优点：

- 进一步减少 RAM 占用
- 适合流式显示

缺点：

- 解码器实现复杂
- 出错处理较难

适用场景：

- `JPEG`、轻量压缩图片
- 有一定软件架构基础的工程

## 7. 推荐的软件架构

如果希望工程结构清晰，建议将显示流程拆成以下几层：

### 7.1 文件访问层

负责：

- `SD` 卡初始化
- 文件系统挂载
- 文件读写接口封装

输出统一接口，例如：

- 打开文件
- 读取文件块
- 定位文件指针

### 7.2 图片识别层

负责：

- 判断是 `BMP`、`JPEG`、`PNG` 还是 `RGB565`
- 提取图片尺寸、位深、像素偏移等信息

### 7.3 图片解码层

负责：

- `BMP` 行解析
- `JPEG` 解码
- `PNG` 解码
- `RGB565` 数据直读

### 7.4 显示输出层

负责：

- 设置显示区域
- 写像素行
- 裁剪超出屏幕部分
- 控制缩放或居中显示

这样分层后，后续新增新格式时，只需补充相应的识别和解码模块，不必改动底层 LCD 驱动。

## 8. 常见问题

### 8.1 RAM 不够怎么办

解决思路：

- 不整图载入
- 改为逐行读取
- 减小行缓冲区
- 优先使用 `BMP` 或 `RGB565`
- 复杂格式放到 PC 端预转换

### 8.2 显示速度太慢怎么办

解决思路：

- 优先使用 `SDIO` 而不是 `SPI`
- 使用 `RGB565` 原始数据减少运行时转换
- 使用 LCD 连续写入接口
- 减少逐点绘制，改为整行或整块写入
- 必要时先缩放或预处理图片

### 8.3 颜色显示不正确怎么办

常见原因：

- `BGR` 与 `RGB` 顺序搞反
- 没有正确转换为 `RGB565`
- 字节高低位顺序错误
- LCD 驱动配置颜色顺序不一致

### 8.4 图片上下颠倒怎么办

常见原因：

- `BMP` 默认可能是倒序存储
- LCD 坐标系方向和图像方向不一致

解决方式：

- 从底行往顶行输出
- 或在显示时调整坐标映射

### 8.5 图片显示越界怎么办

解决思路：

- 在显示前比较图片尺寸与 LCD 分辨率
- 对超出区域进行裁剪
- 或先在 PC 端缩放到目标尺寸

## 9. 面向当前工程的建议方案

结合当前这类 `STM32 + FATFS + LCD` 工程，更推荐以下路线。

### 方案一：BMP 逐行显示

适合：

- 想快速实现
- MCU 资源有限
- 图片数量不多

特点：

- 实现简单
- 调试方便
- 适合作为第一版方案

### 方案二：RGB565 原始文件直显

适合：

- 追求显示效率
- 图片资源固定
- 允许在 PC 端提前转换

特点：

- MCU 侧负担最小
- 显示速度最快
- 最适合产品化方案

### 方案三：JPEG/PNG 运行时解码

适合：

- 平台性能较强
- 需要节省存储空间
- 项目对图片质量和格式兼容性要求较高

特点：

- 灵活性高
- 存储占用较小
- 开发复杂度最高

对一般嵌入式 LCD 项目，推荐优先级通常是：

1. `RGB565`
2. `BMP`
3. `JPEG`
4. `PNG`

这个顺序不是按通用图片格式流行程度排序，而是按 MCU 侧实现难度和运行成本排序。

## 10. 典型实现建议

如果目标是“尽快稳定地从 `SD` 卡显示图片到 `LCD`”，建议采用以下策略：

1. 先实现 `FATFS + BMP` 显示
2. 再实现 `RGB565` 原始文件直显
3. 对大批量图片资源，优先在 PC 端离线转换
4. MCU 端只做最轻量的解析和刷屏

这样做的原因是：

- 风险低
- 调试简单
- 对 RAM 要求小
- 更符合中低端 MCU 的实际能力

## 11. 总结

通过 `SD` 卡在 `LCD` 上显示图片，本质上是“文件读取 + 图片解析 + 像素输出”的组合问题。

从工程角度看：

- `BMP` 最容易从 `SD` 卡直接显示
- `RGB565` 最适合做高效运行时显示
- `JPEG/PNG` 更适合做设计或存储格式，不一定适合在小 MCU 上直接完整解码

对于多数 `STM32` 项目，最务实的路径通常是：

- 开发阶段保留通用图片格式
- 发布阶段转换为 `RGB565` 或简化格式
- 使用 `SD` 卡存储资源
- MCU 逐行或逐块读取并输出到 LCD

这样可以在实现复杂度、显示速度和资源占用之间取得较好的平衡。

## 12. 当前项目中的实际实现说明

结合当前工程，图片显示相关能力已经落地为下面这套行为：

### 12.1 当前支持的格式

当前运行时直接支持：

- `BMP`
- `JPG`
- `JPEG`

其中：

- `BMP` 适合作为最稳定、最容易调试的图片格式
- `JPG/JPEG` 适合作为普通照片类素材的存储格式

### 12.2 当前文件浏览与预览流程

当前工程中的实际流程是：

1. `KEY0` 从主页面进入文件浏览器
2. 浏览器从 `0:/` 根目录开始显示目录和文件
3. 进入任意子目录后，可通过 `[..]` 返回上一级
4. 选中图片文件后按 `KEY2` 进入查看页
5. 如果扩展名是 `.bmp/.jpg/.jpeg`，则直接进入图片预览模式

### 12.3 当前缩放与显示规则

当前预览页面中的图片显示规则如下：

1. 如果图片比预览区域大：
- 按原图比例缩小
- 尽量完整放进预览区域
- 缩小后居中显示

2. 如果图片比预览区域小：
- 不放大
- 保持原尺寸
- 在预览区域内居中显示

3. 每次显示新图片前：
- 先清空预览背景
- 避免上一张图残留像素

### 12.4 BMP 的当前实现方式

当前 `BMP` 路径采用：

- 逐行从 `FATFS` 读取文件
- 解析头信息
- 按需做最近邻缩放
- 转换为 `RGB565`
- 逐行写入 `LCD`

这种方式的优点是：

- RAM 占用较低
- 实现清晰
- 容易调试

### 12.5 JPG/JPEG 的当前实现方式

当前 `JPG/JPEG` 路径采用 `TJpgDec` 运行时解码。

特点：

- 使用软件解码
- 优先利用 `TJpgDec` 自带的缩放级别：
  - `1/1`
  - `1/2`
  - `1/4`
  - `1/8`
- 解码后直接输出到 `LCD`

这种方式可以在当前平台上较轻量地支持 `JPG`，但格式兼容性不是完全通用。

### 12.6 JPG 素材要求

当前工程对 `JPG` 资源更推荐以下规范：

- `Baseline JPEG`
- `RGB`
- 普通导出参数
- 尽量避免复杂编码选项

推荐测试尺寸：

- `320 x 240`
- `240 x 320`
- `240 x 240`

### 12.7 可能出现的 JPG 失败原因

如果某张 `JPG` 打不开，当前状态栏可能显示：

- `JPG UNSUP`
- `JPG NOMEM`
- `JPG FORMAT`

含义分别是：

1. `JPG UNSUP`
- 说明该图片的 JPEG 编码类型超出当前解码器支持范围
- 最常见原因是：
  - `Progressive JPEG`
  - `CMYK JPEG`
  - 特殊采样或特殊编码参数

2. `JPG NOMEM`
- 说明当前 JPEG 解码工作区内存不足

3. `JPG FORMAT`
- 说明 JPEG 文件结构损坏或头信息不合法

### 12.8 当前推荐的工程使用策略

对于当前这套 `STM32F407 + FATFS + LCD` 工程，推荐这样使用图片：

1. 调试和验证阶段：
- 优先使用 `24-bit BMP`
- 便于观察颜色、方向、缩放和显示链路

2. 照片类素材：
- 使用 `Baseline JPG`
- 统一导出参数

3. 如果后续需要更高稳定性或更高显示效率：
- 继续转向 `RGB565` 这类 MCU 友好格式
- 把复杂的图像处理尽量放在 PC 端完成
