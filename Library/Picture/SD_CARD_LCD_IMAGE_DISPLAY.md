# SD Card LCD Image Display

## 1. 概述

在嵌入式系统中，通过 `SD` 卡在 `LCD` 屏幕上显示图片，通常不是简单地“读取文件然后显示”，而是一个完整的数据处理链路：

```text
SD 卡
  -> 文件系统读取
  -> 图片格式识别
  -> 图片数据解析或解码
  -> 像素格式转换
  -> LCD 刷屏显示
```

在这个流程中，真正送到 `LCD` 的并不是 `BMP`、`JPEG`、`PNG` 这些文件本身，而是已经整理成屏幕可以识别的像素流，例如 `RGB565`。

因此，使用 `SD` 卡显示图片时，需要同时解决以下几个问题：

- 如何从 `SD` 卡读取图片文件
- 如何识别图片类型
- 如何把图片数据转换为 LCD 需要的像素格式
- 如何在 RAM 很小的情况下完成显示

## 2. 系统组成

一个典型的 `STM32 + SD 卡 + LCD` 显示系统，通常由以下几个部分组成。

### 2.1 SD 卡接口层

`SD` 卡通常通过以下方式连接：

- `SDIO` / `SDMMC`
- `SPI`

其中：

- `SDIO/SDMMC` 速度较高，更适合图片和文件频繁读取
- `SPI` 接线简单，但吞吐量通常更低

如果图片较大，或者需要切换多张图片，优先考虑 `SDIO`。

### 2.2 文件系统层

大多数嵌入式工程会使用 `FATFS` 读取 `SD` 卡中的文件。

常见操作包括：

- 挂载文件系统
- 打开图片文件
- 读取文件头
- 按块或按行读取数据

例如：

- `f_mount()`
- `f_open()`
- `f_read()`
- `f_lseek()`
- `f_close()`

### 2.3 图片解析层

这一层负责判断当前文件属于哪种格式，并执行相应处理：

- `BMP`：读取文件头、宽高、位深，按行读取像素
- `JPEG`：调用 JPEG 解码器
- `PNG`：调用 PNG 解码器
- `RGB565/RAW`：按预定义宽高和像素格式直接读取

### 2.4 LCD 驱动层

这一层负责将像素输出到屏幕，常见操作包括：

- 设置显示窗口
- 写入单个像素
- 连续写入一行像素
- 整块区域刷新

对于彩色 LCD，通常目标格式为：

- `RGB565`

因此，很多显示流程的核心工作就是“把文件中的图像数据变成 `RGB565`”。

## 3. 通过 SD 卡显示图片的整体流程

一般可按以下步骤实现：

1. 初始化 `SD` 卡接口
2. 挂载 `FATFS`
3. 打开指定图片文件
4. 根据文件头或扩展名判断图片类型
5. 解析图片宽度、高度、像素深度
6. 选择合适的解码方式
7. 逐行或逐块读出像素
8. 转换为 `LCD` 所需格式
9. 写入 LCD 显示区域
10. 文件读取结束后关闭文件

如果 MCU 内存有限，不建议一次性将整张图片全部读入内存，而应采用：

- 逐行读取
- 分块读取
- 边读边解码边显示

## 4. 各种常见图片类型的实现路径

不同图片格式在嵌入式中的处理方式差异很大。核心区别在于：文件结构是否简单，运行时解码开销是否可接受。

### 4.1 BMP 图片

`BMP` 是在嵌入式中最容易从 `SD` 卡读取并显示的格式之一。

原因：

- 文件头结构清晰
- 宽高、位深容易获取
- 支持逐行读取
- 不依赖复杂压缩解码

典型流程：

1. 打开 `BMP` 文件
2. 读取文件头和信息头
3. 获取图片宽度、高度、位深、像素数据偏移
4. 根据位深选择处理方式，例如：
   - `16bit BMP`
   - `24bit BMP`
   - `32bit BMP`
5. 定位到像素起始位置
6. 每次读取一行像素
7. 将像素转换为 `RGB565`
8. 写入 LCD

需要注意：

- 很多 `BMP` 采用“自底向上”存储
- 每行可能带字节对齐填充
- `24bit BMP` 通常是 `BGR` 排列，不是 `RGB`

优点：

- 实现简单
- 适合小内存 MCU
- 非常适合从 `SD` 卡逐行读取显示

缺点：

- 文件体积通常较大
- 不适合大量照片资源长期存储

### 4.2 JPEG 图片

`JPEG` 适合存储照片类图片，优点是体积小，但在 MCU 上直接显示时，问题主要在解码复杂度。

典型流程：

1. 从 `SD` 卡读取 `JPEG` 文件
2. 调用 JPEG 解码库解析压缩数据
3. 输出像素块或行缓冲
4. 转换为 `RGB565`
5. 刷新到 LCD

优点：

- 文件小，节省 `SD` 卡空间
- 适合相册、照片浏览

缺点：

- 解码需要较高 CPU 性能
- 需要较大工作缓冲区
- 软件实现复杂度高

适用场景：

- `STM32F4/F7/H7` 等性能较强平台
- 带足够 RAM 的系统
- 已集成 JPEG 解码库或硬件加速单元

对于普通小型 MCU，如果只是显示固定图片，不建议运行时直接解码 `JPEG`，更实用的做法通常是：

- 在 PC 端先转成 `RGB565`
- 再存入 `SD` 卡供 MCU 直接读取显示

### 4.3 PNG 图片

`PNG` 常用于图标、界面素材、透明背景资源。但在 MCU 上直接从 `SD` 卡读取并显示时，难点在于：

- 存在无损压缩
- 可能有 Alpha 通道
- 解码步骤多于 `BMP`

典型流程：

1. 从 `SD` 卡读取 `PNG` 文件
2. 调用 PNG 解码库展开像素数据
3. 必要时处理透明通道
4. 转换成 `RGB565`
5. 刷新 LCD

优点：

- 适合 UI 图标与带透明背景素材
- 文件通常比 BMP 更小

缺点：

- 解码复杂
- RAM 压力较大
- 透明混合会增加运算量

适用场景：

- 带 GUI 框架的较强平台
- 对 UI 效果要求较高的系统

对大部分 `STM32 + 普通 LCD` 工程，更推荐：

- 设计阶段用 `PNG`
- 发布前离线转成 `RGB565` 或轻量压缩格式

### 4.4 RGB565 / RAW 图片

这类格式不是通用桌面图片格式，而是面向嵌入式显示优化的原始像素数据。

常见做法：

- 在 PC 端把图片转换为 `RGB565`
- 存成二进制文件放入 `SD` 卡
- MCU 运行时直接读出并写入 LCD

典型流程：

1. 打开 `RGB565` 原始文件
2. 已知图片宽高
3. 每次从 `SD` 卡读取一行或若干像素
4. 直接送到 LCD

优点：

- 显示最快
- MCU 逻辑最简单
- 很适合逐行显示

缺点：

- 文件不自描述，通常需要额外记录宽高和格式
- 文件体积比压缩格式大

这是嵌入式中最务实、最常见的 `SD` 卡图片显示方案之一。

## 5. 小内存 MCU 的三种显示策略

在 MCU RAM 很有限的情况下，显示图片时通常不能把整图完整载入内存。常见策略如下。

### 5.1 整图载入

方式：

- 整个文件读入 RAM
- 解析后一次性显示

优点：

- 控制简单
- 适合小图标和缩略图

缺点：

- 大图容易耗尽 RAM

适用场景：

- 小尺寸图片
- 系统 RAM 较大

### 5.2 逐行读取

方式：

- 每次只读取一行像素数据
- 转换后立即写入 LCD

优点：

- RAM 占用小
- 非常适合 `BMP` 和 `RGB565`

缺点：

- 文件读取次数较多
- 程序控制略复杂

适用场景：

- 中低端 MCU
- 中大尺寸图片显示

### 5.3 边读边解码边显示

方式：

- 从 `SD` 卡读一部分压缩数据
- 解码出像素后立即刷新 LCD

优点：

- 进一步减少 RAM 占用
- 适合流式显示

缺点：

- 解码器实现复杂
- 出错处理较难

适用场景：

- `JPEG`、轻量压缩图片
- 有一定软件架构基础的工程

## 6. 推荐的软件架构

如果希望工程结构清晰，建议将显示流程拆成以下几层：

### 6.1 文件访问层

负责：

- `SD` 卡初始化
- 文件系统挂载
- 文件读写接口封装

输出统一接口，例如：

- 打开文件
- 读取文件块
- 定位文件指针

### 6.2 图片识别层

负责：

- 判断是 `BMP`、`JPEG`、`PNG` 还是 `RGB565`
- 提取图片尺寸、位深、像素偏移等信息

### 6.3 图片解码层

负责：

- `BMP` 行解析
- `JPEG` 解码
- `PNG` 解码
- `RGB565` 数据直读

### 6.4 显示输出层

负责：

- 设置显示区域
- 写像素行
- 裁剪超出屏幕部分
- 控制缩放或居中显示

这样分层后，后续新增新格式时，只需补充相应的识别和解码模块，不必改动底层 LCD 驱动。

## 7. 常见问题

### 7.1 RAM 不够怎么办

解决思路：

- 不整图载入
- 改为逐行读取
- 减小行缓冲区
- 优先使用 `BMP` 或 `RGB565`
- 复杂格式放到 PC 端预转换

### 7.2 显示速度太慢怎么办

解决思路：

- 优先使用 `SDIO` 而不是 `SPI`
- 使用 `RGB565` 原始数据减少运行时转换
- 使用 LCD 连续写入接口
- 减少逐点绘制，改为整行或整块写入
- 必要时先缩放或预处理图片

### 7.3 颜色显示不正确怎么办

常见原因：

- `BGR` 与 `RGB` 顺序搞反
- 没有正确转换为 `RGB565`
- 字节高低位顺序错误
- LCD 驱动配置颜色顺序不一致

### 7.4 图片上下颠倒怎么办

常见原因：

- `BMP` 默认可能是倒序存储
- LCD 坐标系方向和图像方向不一致

解决方式：

- 从底行往顶行输出
- 或在显示时调整坐标映射

### 7.5 图片显示越界怎么办

解决思路：

- 在显示前比较图片尺寸与 LCD 分辨率
- 对超出区域进行裁剪
- 或先在 PC 端缩放到目标尺寸

## 8. 面向当前工程的建议方案

结合当前这类 `STM32 + FATFS + LCD` 工程，更推荐以下路线。

### 方案一：BMP 逐行显示

适合：

- 想快速实现
- MCU 资源有限
- 图片数量不多

特点：

- 实现简单
- 调试方便
- 适合作为第一版方案

### 方案二：RGB565 原始文件直显

适合：

- 追求显示效率
- 图片资源固定
- 允许在 PC 端提前转换

特点：

- MCU 侧负担最小
- 显示速度最快
- 最适合产品化方案

### 方案三：JPEG/PNG 运行时解码

适合：

- 平台性能较强
- 需要节省存储空间
- 项目对图片质量和格式兼容性要求较高

特点：

- 灵活性高
- 存储占用较小
- 开发复杂度最高

对一般嵌入式 LCD 项目，推荐优先级通常是：

1. `RGB565`
2. `BMP`
3. `JPEG`
4. `PNG`

这个顺序不是按通用图片格式流行程度排序，而是按 MCU 侧实现难度和运行成本排序。

## 9. 典型实现建议

如果目标是“尽快稳定地从 `SD` 卡显示图片到 `LCD`”，建议采用以下策略：

1. 先实现 `FATFS + BMP` 显示
2. 再实现 `RGB565` 原始文件直显
3. 对大批量图片资源，优先在 PC 端离线转换
4. MCU 端只做最轻量的解析和刷屏

这样做的原因是：

- 风险低
- 调试简单
- 对 RAM 要求小
- 更符合中低端 MCU 的实际能力

## 10. 总结

通过 `SD` 卡在 `LCD` 上显示图片，本质上是“文件读取 + 图片解析 + 像素输出”的组合问题。

从工程角度看：

- `BMP` 最容易从 `SD` 卡直接显示
- `RGB565` 最适合做高效运行时显示
- `JPEG/PNG` 更适合做设计或存储格式，不一定适合在小 MCU 上直接完整解码

对于多数 `STM32` 项目，最务实的路径通常是：

- 开发阶段保留通用图片格式
- 发布阶段转换为 `RGB565` 或简化格式
- 使用 `SD` 卡存储资源
- MCU 逐行或逐块读取并输出到 LCD

这样可以在实现复杂度、显示速度和资源占用之间取得较好的平衡。
