# Picture

## 1. 常见图片格式简介

在通用计算机和移动设备中，图片通常以文件格式的形式存在，例如 `BMP`、`JPEG`、`PNG`、`GIF`、`WebP` 等。这些格式既决定了图片如何存储，也决定了图片在读取时需要怎样解析。

### 1.1 BMP

- `BMP` 是最典型的位图格式。
- 结构相对简单，通常包含文件头、信息头和像素数据。
- 可以是未压缩存储，也可以带少量简单压缩。
- 优点是解析简单，适合初学和资源较少的系统。
- 缺点是体积通常较大，不利于在 Flash、SD 卡或串口链路中传输。

### 1.2 JPEG

- `JPEG` 是常见的有损压缩图片格式。
- 适合照片、复杂颜色渐变场景。
- 压缩率高，文件体积小。
- 缺点是解码过程复杂，需要较多运算和缓存。
- 在 MCU 上直接解码 `JPEG`，通常需要更高主频、更大 RAM，或者借助专用硬件。

### 1.3 PNG

- `PNG` 是常见的无损压缩图片格式。
- 支持透明通道，适合图标、界面元素、线条图。
- 画质稳定，重复解码不会劣化。
- 但其解码仍然比原始位图复杂，特别是在 RAM 很小的单片机场景中。

### 1.4 GIF

- `GIF` 支持有限颜色表和简单动画。
- 常见于小尺寸动图。
- 颜色数有限，不适合高质量静态图片显示。
- 在嵌入式 LCD 中偶尔用于简单动画演示，但不是主流方案。

### 1.5 WebP

- `WebP` 是现代图片格式，可支持有损和无损压缩。
- 在桌面和移动端使用越来越多。
- 对嵌入式 MCU 来说，编解码复杂度通常偏高。
- 除非系统性能较强，否则不常直接作为 LCD 显示的运行时格式。

### 1.6 RAW / 原始像素数据

- 严格来说，`RAW` 不一定是统一标准格式，更像“没有复杂封装的原始像素数组”。
- 常见形式包括 `RGB565`、`RGB888`、`ARGB8888`、灰度图等。
- 优点是解码几乎不需要额外步骤，只要知道宽高、像素格式和存储顺序即可直接显示。
- 缺点是体积大，缺少通用文件头与压缩能力。

## 2. 嵌入式 LCD 为什么更关心像素格式

在 PC 上，图片常以“文件格式”为中心；而在嵌入式 LCD 系统中，真正送到屏幕上的通常不是 `PNG` 或 `JPEG` 文件本身，而是已经展开后的像素数据。

LCD 控制器更关心的是：

- 每个像素占多少位
- 颜色分量如何排列
- 一行像素如何连续存储
- 数据是从左到右还是从上到下输出
- 是否需要写入显存，还是直接逐点刷新

因此，在嵌入式开发中，经常要区分两个层面：

1. 存储层：图片文件如何保存在 Flash、SD 卡或外部存储器中
2. 显示层：LCD 最终需要的像素流格式是什么

很多时候，系统会在 PC 端先把通用图片转换成适合 LCD 的格式，再放入工程中使用。

## 3. LCD 常见像素数据格式

### 3.1 RGB565

`RGB565` 是嵌入式彩色 LCD 中最常见的像素格式之一。

- 红色 5 位
- 绿色 6 位
- 蓝色 5 位
- 共 16 位，即 2 字节

优点：

- 存储空间较省
- 与很多 TFT LCD 控制器兼容良好
- MCU 运算和传输负担比 `RGB888` 小

缺点：

- 颜色精度不如 24 位真彩

### 3.2 RGB888

- 每个像素 24 位，红绿蓝各 8 位
- 颜色效果更好
- 但内存占用和带宽要求更高

如果 MCU RAM 较小，通常不适合整屏缓存 `RGB888`。

### 3.3 ARGB8888 / RGBA8888

- 在 `RGB888` 基础上增加透明通道
- 常用于 GUI 框架、上位机图形系统
- 对普通 MCU + LCD 场景来说，存储和运算开销较大

### 3.4 灰度图 / 单色位图

- 单色屏或低资源系统中常见
- 每个像素可能是 1 位、2 位、4 位或 8 位
- 适用于字库、图标、简单 UI 元素

## 4. 嵌入式中的图像编解码是什么意思

在嵌入式 LCD 场景里，图像“编码”和“解码”通常不是指完整桌面图像软件那种复杂处理，而是指：

- 编码：把通用图像转换成适合目标 LCD 的存储格式
- 解码：把存储格式还原成 LCD 可以直接显示的像素流

### 4.1 编码

编码往往发生在 PC 端或资源生成工具中，例如：

- 将 `PNG` 转换成 `RGB565` 数组
- 将 `BMP` 转换成 C 数组
- 将图像切片、量化、压缩
- 生成字模或图标表

这一步的目的，是把“易于编辑的格式”变成“易于 MCU 显示的格式”。

### 4.2 解码

解码发生在 MCU 运行时，常见形式包括：

- 直接读取 `RGB565` 数据并写入 LCD
- 从外部 Flash 中读取压缩图像，再边解码边显示
- 从 SD 卡读取 `BMP`，解析文件头后逐行输出到 LCD
- 对 RLE 等轻量压缩格式进行展开

## 5. 在嵌入式 LCD 中常见的图像处理流程

典型流程如下：

```text
PC 上的原始图片
    -> 图片转换工具
    -> 生成适合 MCU 的图像数据
    -> 存入工程数组 / 外部 Flash / SD 卡
    -> MCU 读取数据
    -> 如有需要进行解码或像素格式转换
    -> 写入 LCD 控制器或显存
```

这个流程中，真正影响性能的关键点通常有三个：

- 存储空间是否足够
- RAM 是否足够做缓存
- LCD 接口带宽是否足够

## 6. 常见图像文件在嵌入式中的使用方式

### 6.1 BMP 的使用方式

`BMP` 常被当作“容易处理的中间格式”：

- 文件头简单
- 可直接读取宽高和位深
- 如果是 16 位或 24 位 BMP，比较容易转换到 LCD 像素格式

常见做法：

- PC 端把其他格式先转成 BMP
- MCU 端解析 BMP 文件头
- 按行读取像素，转换后输出到 LCD

适合：

- 演示程序
- SD 卡读图
- 对解码复杂度敏感的场景

### 6.2 JPEG 的使用方式

`JPEG` 更适合存储照片，但 MCU 端直接解码代价较高。

常见做法：

- 在 PC 端将 JPEG 转换为 `RGB565` 数组后再编进固件
- 在性能较强的平台上使用软件 JPEG 解码库
- 借助带图像加速或硬件解码能力的芯片

适合：

- 存储空间紧张
- 图片以照片为主
- 平台具备足够运算能力

### 6.3 PNG 的使用方式

`PNG` 常用于图标、按钮、UI 资源。

常见做法：

- 开发阶段使用 PNG 方便设计
- 发布前离线转换为 `RGB565`、带透明色的数组或轻量压缩格式

适合：

- UI 图标
- Logo
- 需要透明背景的资源设计阶段管理

### 6.4 RAW / RGB565 的使用方式

这是嵌入式中最直接、最常用的运行时格式之一。

常见做法：

- 用上位机工具将图片转成 `RGB565`
- 以数组形式编译进程序
- 或保存在外部 Flash/SD 卡中，运行时直接读取

适合：

- 资源较固定
- 追求显示速度
- MCU 性能有限

## 7. 嵌入式 LCD 中常见的编码方案

### 7.1 直接数组编码

把图像转换成 C 数组，例如：

```c
const unsigned short image_data[] = {
    0xFFFF, 0xF800, 0x07E0, 0x001F
};
```

特点：

- 使用最简单
- 显示速度快
- 适合小图标、开机画面、固定背景

缺点：

- 占用固件空间
- 修改图片需要重新生成并重新编译

### 7.2 外部存储编码

将图像保存在：

- SPI Flash
- NOR Flash
- NAND Flash
- SD 卡

运行时再读取。

特点：

- 节省内部 Flash
- 适合大量图片资源
- 但读取速度受接口影响

### 7.3 轻量压缩编码

在 MCU 场景中，常见的是比 `JPEG/PNG` 更轻量的压缩方式，例如：

- 行程编码 `RLE`
- 按块存储
- 调色板索引压缩
- 针对单色/灰度图的位打包

特点：

- 解码逻辑相对简单
- 压缩率通常不如 JPEG/PNG
- 更适合 MCU 在线解码

## 8. 嵌入式 LCD 中常见的解码方案

### 8.1 无解码直显

如果资源已经是 `RGB565` 且顺序正确，那么 MCU 只需：

- 定位显示区域
- 循环发送像素数据

这是效率最高、实现最简单的方案。

### 8.2 文件头解析后逐行显示

例如显示 `BMP` 文件时，常见步骤是：

1. 读取文件头
2. 解析宽度、高度、位深
3. 逐行读取像素
4. 转换为 LCD 目标格式
5. 写入 LCD

好处是：

- 不必整图载入 RAM
- 适合内存较小的单片机

### 8.3 边读边解压边显示

对于 `RLE` 或其他轻量压缩图像，可采用流式方式：

1. 从外部存储读一段压缩数据
2. 解码出若干像素
3. 立即送往 LCD

这种方法可以减少 RAM 占用，但实现上要注意：

- 行边界处理
- 解码状态保存
- 显示窗口坐标控制

## 9. 嵌入式系统中选择图片格式时需要考虑什么

选择格式时，一般要综合考虑以下因素：

### 9.1 存储空间

- 内部 Flash 很小时，不适合大量原始位图
- 外部 Flash 或 SD 卡充足时，可以接受较大资源文件

### 9.2 RAM 大小

- 如果 RAM 很小，应避免整图解码
- 更适合逐行读取、逐块显示

### 9.3 MCU 算力

- 低端 MCU 更适合 `RGB565` 直显或轻量解码
- 高性能 MCU 才更适合复杂格式在线解码

### 9.4 LCD 接口带宽

- SPI 屏幕刷新较慢，适合小图、局部刷新、预处理后的像素数据
- FSMC、8080 并口、RGB 接口带宽更高，更适合大图和频繁刷新

### 9.5 图片类型

- 照片类：更偏向 JPEG 存储
- 图标类：更偏向 PNG 设计、RGB565/RLE 运行
- 单色字库：更偏向位图压缩或点阵存储

## 10. 面向嵌入式 LCD 的典型建议

### 10.1 小型 MCU + 普通 TFT LCD

推荐：

- 开发阶段使用 PNG/BMP 作为素材
- 发布前统一转换为 `RGB565`
- 小图直接编译进数组
- 大图存放在外部 Flash 或 SD 卡

### 10.2 需要大量界面图标

推荐：

- 素材用 PNG 管理
- 运行时使用 `RGB565` 或调色板压缩格式
- 对透明区域可以使用透明色键值或单独 Alpha 掩码

### 10.3 需要显示照片

推荐：

- 存储时使用 JPEG 节省空间
- 如果 MCU 算力不足，则由上位机预转换
- 如果平台足够强，再考虑运行时 JPEG 解码

## 11. 与当前工程相关的实践建议

结合当前这类 STM32 + LCD 工程，通常更推荐如下路线：

1. 设计素材阶段使用 `PNG` 或 `BMP`
2. 使用上位机工具将图片转换为 `RGB565`
3. 小图片转成 C 数组放入工程
4. 大图片放入外部 Flash 或 SD 卡
5. MCU 端尽量只做轻量格式解析，不做复杂图像解码

原因是：

- `RGB565` 与多数 LCD 库配合最直接
- MCU 侧逻辑简单
- 调试难度低
- 刷屏速度更稳定

如果后续需要扩展图像模块，可以继续按以下方向演进：

- 增加 `BMP` 文件读取与逐行显示
- 增加 `RGB565` 资源打包格式
- 增加 `RLE` 轻量压缩与解码
- 为 UI 图标增加透明色或蒙版机制

## 12. 总结

在嵌入式 LCD 系统中，图片文件格式只是资源输入方式，真正决定显示效率的核心是像素格式、存储方式和解码复杂度。

对于大多数 MCU 项目来说，最务实的策略是：

- 用 `PNG/JPEG/BMP` 作为设计和交换格式
- 用 `RGB565` 或轻量压缩格式作为运行时显示格式
- 尽量把复杂编码工作放在 PC 端完成
- 让 MCU 专注于读取、轻量解码和 LCD 刷新

这样可以在存储空间、显示速度和实现复杂度之间取得较好的平衡。
